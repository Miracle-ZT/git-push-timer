## 3. 第一轮全项目代码 Review 结论

以下问题来自对开发仓库代码的静态 review、行为验证和文档比对。

### 3.1 高优先级问题

1. `os.Chdir` 修改进程全局工作目录，和 cron 并发 job 组合后存在目录竞争风险。
   - 文件：`internal/executor/executor.go`
   - 风险：多个仓库同时触发时，Git 命令可能在错误目录执行，或者互相踩锁。

2. 变更检测用 `git diff --quiet HEAD`，不会把未跟踪文件算作变更。
   - 文件：`internal/executor/executor.go`
   - 风险：新建文件会被误判为“无变更”，永远不会被自动提交。

3. 同一处变更检测把“真的有变更”和“Git 命令失败”混在一起处理。
   - 文件：`internal/executor/executor.go`
   - 风险：例如仓库没有首个提交时，`git diff --quiet HEAD` 返回 `128`，当前逻辑会把它当成“有变更”继续往下执行。

### 3.2 中优先级问题

4. 文档宣称“启动后立即执行一次检查”，实现并没有这样做。
   - 文件：`internal/scheduler/scheduler.go`
   - 文档：`README.md`、`DEVELOPMENT.md`

5. 退出流程没有等待正在执行的任务安全收尾。
   - 文件：`cmd/git-push-timer/main.go`
   - 风险：`git commit` / `git push` 期间如果退出，可能被中断。

### 3.3 后续排查中新增的问题

6. macOS 睡眠后，长周期 cron 实际按累计可运行时间推进，导致日级任务严重漂移或看起来不触发。

7. 日志文件没有按自然日轮转，而是按“进程启动当日”固定写入同一个文件。

### 5.2 再确认是不是 cron 表达式本身有问题

观察到 release 目录里旧日志：

- 文件：`/Users/tgb/Documents/git-push-timer/logs/2026-04-08.log`
- 内容显示该 same release 目录上的 `00 18 * * *` 曾在 `2026-04-08 18:00:00` 正常触发并成功推送

结论：

- `00 18 * * *` 这个 cron 表达式本身没有问题
- 问题更可能与“进程持续运行 + 睡眠/唤醒 + 长周期等待”有关

### 5.3 再看 robfig/cron 的实际实现

`robfig/cron/v3@v3.0.1` 的关键逻辑：

1. 启动时先用 `Schedule.Next(now)` 算出每个任务的下次执行时间
2. 然后直接调用：

```go
timer = time.NewTimer(c.entries[0].Next.Sub(now))
```

也就是说：

- 它不是“每分钟轮询一次现在是否到了 18:00”
- 它是“一次性计算出距离下次执行还有多久，然后睡这么久”

这对短周期任务通常没问题，但对“下一次执行要等将近 24 小时”的场景，睡眠行为会变得非常关键。

### 5.4 再看 Go 在 macOS 上 timer 依赖什么时钟

检查了 Go 1.21 的 darwin runtime：

- `runtime` 的 `nanotime` 最终调用 `mach_absolute_time()`

检查了 Apple 的头文件 `mach_time.h`：

- `mach_continuous_time()` 的注释明确写着：
  - `like mach_absolute_time, but advances during sleep`

这反向说明：

- `mach_absolute_time()` 本身不在睡眠期间推进
- 基于它的 timer 也就不会按墙上时间连续推进

结论：

- 当前这套 `robfig/cron + time.NewTimer + Go darwin runtime` 组合，在 macOS 睡眠期间，不会把“睡过去的墙上时间”完整计入长 timer

## 6. 为什么 `*/10` 看起来能恢复，而 daily 不行

之前有一个对照现象：

- 开发目录里用 `*/10 * * * *` 跑过一次
- 用户印象是“短频率任务睡醒后能继续执行”

复查 `2026-04-13.log` 后发现，更准确的描述应该是：

- 它并不是“一直睡死，直到第二天醒来才恢复一次”
- 它是在整晚多次 `DarkWake` / `FullWake` 过程中，断断续续地多次触发了任务
- 日志中存在大量跨夜执行记录，例如：
  - `2026-04-13 20:06:53`
  - `2026-04-13 20:10:00`
  - `2026-04-13 20:20:00`
  - ...
  - `2026-04-14 09:21:37`

因此更合理的解释是：

- `*/10` 这种短 timer，只需要累计很短的可运行时间
- 机器夜里有大量 `DarkWake` 维护唤醒窗口，足够让短 timer 一次次完成
- `00 18 * * *` 这种接近 24 小时的长 timer，需要累计太多可运行时间，睡眠会让它严重拖后

## 9. 当前最可信的主假设

### 假设内容

`robfig/cron` 为 daily 任务创建了一个约 `23小时56分09秒` 的长 timer。

在 macOS 上，这个 timer 依赖的底层时间推进不包含正常睡眠时长，因此：

- 机器虽然从墙上时间看已经过了 1 天多
- 但进程实际累计到的“timer 可推进时间”只有 `13小时48分22秒`
- 所以任务尚未触发

### 该假设解释了什么

它能解释：

1. 为什么 startup 日志正常，但后面一直没有 `触发仓库：Bruno-TGB`
2. 为什么进程还活着，却没有任何后续动作
3. 为什么 `*/10` 这种短周期任务看起来“睡醒后能继续”，而 daily 这种长周期任务会严重漂移

### 该假设还不能 100% 证明什么

它目前是“最符合现有证据的解释”，但还没有最终闭环验证。

真正闭环的验证标准是：

- 如果后续在累计可运行时间超过 `23小时56分09秒` 后，这个老进程真的触发了任务，那么该假设会被显著增强
- 如果累计明显超过该阈值，进程仍然毫无动静，则说明还存在别的原因

## 10. 第二个独立问题：日志并未按自然日切分

这个问题在排查过程中被确认。

当前 logger 的行为是：

1. 进程启动时计算一次 `today := time.Now().Format("2006-01-02")`
2. 用这个日期打开日志文件
3. 之后始终往这个文件句柄写

因此：

- `2026-04-14` 启动的进程
- 即使在 `2026-04-15` 或 `2026-04-16` 执行了任务
- 也仍然会继续写入 `logs/2026-04-14.log`

这意味着：

- “没有 `2026-04-15.log` / `2026-04-16.log`” 不能直接推出“那两天没执行”
- 回头复盘这个老进程时，应该先看 `2026-04-14.log`

注意：

- 这个问题会放大排查难度
- 但它本身**不能单独解释**“任务为什么没有触发”
- 它只是说明“即使触发了，也可能写在旧文件里”

#### 粗粒度统计

如果只按这些边界事件统计：

- `Sleep Entering Sleep state`
- `DarkWake from ...`
- `Wake from ...`

1. 每次唤醒本身的 `WakeTime`

- 共出现 `58` 次
- 合计约 `790.59` 秒

把这部分加回去后：

- `23:38:02 + 13:10.59 = 23:51:12.59`

距离理论值 `23:56:09` 只差：

- 约 `4分56秒`

而这剩下的几分钟，完全可能来自：

- 进入睡眠前的过渡延迟
- `pmset` 日志边界本身不是为“精确模拟 Go timer 计时”设计的
- 若干状态切换中的记录时差

### 16.4 截至 2026-04-17 的阶段性结论

到这里可以把结论升级为：

- `macOS 睡眠后，长周期 cron 不按墙上时间触发，而是被明显拖延`
- 这个问题已经被**强证据支持**
- 严格说仍然不是“数学证明”，但已经非常接近实证成立

相比之前“只是一个合理猜想”，现在更准确的表述应该是：

- 新触发记录明显站在这一解释一边

### 17.4 这次排查中一个重要的措辞修正

在中途分析时，曾出现过两种说法：

#### 正确说法

- “程序内部那个本该在 `15:02:00` 左右执行的 `60s` 轮询，直到 `15:30:48` 才真正被恢复执行”

#### 不准确说法

- “睡眠导致原本该在 `15:20` 之后继续分钟轮询的那个 `60s` timer 被拖延”

第二种说法不准确，原因是：

- 程序在 `15:00:00` 触发后，会把 `nextRun` 推到 `15:20:00`
- 但内部短轮询本身仍然是每分钟跑一次
- 真正被睡眠打断的，不是 `15:20` 之后的某个轮询
- 而是睡前已经排好的那次“下一分钟检查”

也就是：

- `15:00:00` 正常执行
- `15:01:00` 左右程序大概率做过一次轮询检查，但因为 `now < 15:20:00`，所以没有日志
- 这次轮询结束后，本来应在 `15:02:00` 左右进入下一轮检查
- 但机器在 `15:01:56` 已经睡眠
- 所以“本该在 `15:02:00` 跑到的那次 60 秒轮询”，直到 `15:30:48` 才真正恢复执行

这也是为什么：

- 不是 `15:20` 才开始出问题
- 而是从 `15:02` 开始，这个短 timer 就已经被冻结了

### 17.5 这说明当前修复的本质是什么

从这次复盘可以更准确地定义当前方案：

- 它并没有彻底消灭“进程内 timer 在 macOS 睡眠下被拖延”的问题
- 它只是把原来“可能长达数小时或接近 24 小时的超长 timer 漂移”压缩成了“60 秒级短 timer 的唤醒抖动”

所以可以这样理解：

1. 原方案的问题本质仍然存在
2. 但问题窗口已经从“超长等待”缩小成“短轮询恢复误差”
3. 对实际使用来说，影响已经小很多

### 17.6 当前方案在这类场景下的实际表现

通过这次日志，当前方案的行为边界已经很清楚：

#### 已经达到的目标

- 睡眠错过多个时间点时，只补跑 1 次
- 补跑后不会继续漂移到几个小时后
- 补跑后会重新回到规则的 cron 节奏

这次的实际例子就是：

- 错过 `15:20:00`
- 在 `15:30:48` 补跑 1 次
- 然后重新回到：
  - `15:40:00`
  - `16:00:00`

#### 尚未达到的更高目标

- 唤醒后第一次补跑不一定恰好贴近“最近的整分钟”
- 第一次补跑仍然可能晚几十秒

本例中：

- Full Wake：`15:30:04`
- 首次补跑：`15:30:48`
- 误差约 `44s`

### 17.7 截至本次复盘的结论

这次例子说明：

1. 当前修复已经基本解决了“长时间漂移”的核心问题
2. 当前仍保留一个较小的剩余问题：
   - 睡眠唤醒后的第一次补跑，仍然受进程内短 timer 恢复时机影响
3. 这个剩余问题的性质，和旧问题是同类的，但量级已经从“小时级”降到了“秒级到一分钟内”

因此，对当前方案最准确的总结是：

- 它不是“彻底免疫睡眠影响”
- 而是“把睡眠影响从不可接受的大漂移，压缩到通常可接受的小抖动”

### 17.8 如果后续还要继续优化

若未来希望把这几十秒也尽量消除，下一步的方向不是再调整 cron 计算，而是：

- 在检测到系统唤醒事件时
- 主动立即调用一次 `checkDueRepositories()`
- 而不是只等待程序内部的下一次短 timer 恢复

这样第一次补跑才会更贴近：

- `Wake from Normal Sleep`

而不是像这次一样，拖到：

- `15:30:48`

### 17.9 复盘速查：后续再看到类似日志时怎么判断

为了避免后续再次把现象理解偏，这里把本轮排查里最关键的两类“日志模式 -> 解释”固定下来。

#### 模式 A：老版 long timer 漂移

实际样例：

```text
[2026-04-14 18:03:51] INFO Git Push Timer v1.0.0
...
[2026-04-17 10:06:34] INFO 触发仓库：Bruno-TGB
```

对应解释：

- 这类现象说明原来那个接近 `24h` 的进程内长 timer 被 macOS 睡眠严重拖后
- 它不是“到了 `2026-04-15 18:00:00` 却没命中一次整点轮询”
- 而是“连下一次真正到期的内部等待本身都没有按墙上时间走完”

#### 模式 B：修复后第一次补跑有几十秒抖动

实际样例：

程序日志：

```text
[2026-04-17 14:59:50] INFO Git Push Timer dev
[2026-04-17 15:00:00] INFO 触发仓库：Bruno-TGB
[2026-04-17 15:30:48] INFO 触发仓库：Bruno-TGB
[2026-04-17 15:40:00] INFO 触发仓库：Bruno-TGB
[2026-04-17 16:00:00] INFO 触发仓库：Bruno-TGB
```

系统日志：

```text
2026-04-17 15:01:56 +0800 Sleep                Entering Sleep state due to 'Software Sleep pid=155'
2026-04-17 15:29:52 +0800 Notification         Display is turned on
2026-04-17 15:30:04 +0800 Wake                 Wake from Normal Sleep [CDNVA] : due to XHC2 XHC3 XHC1/UserActivity Assertion Using AC (Charge:0%)
2026-04-17 15:30:49 +0800 Assertions           PID 94(powerd) TimedOut InternalPreventSleep "com.apple.powermanagement.acwakelinger" 00:00:44
```

对应解释：

- `15:00:00` 这次任务完成后，仓库的 `nextRun` 被推进到 `15:20:00`
- 但真正被睡眠打断的，不是“`15:20` 之后的轮询”
- 而是程序内部那个本该在 `15:02:00` 左右执行的下一次 `60s` 轮询
- 它一直等到机器恢复、timer 真正重新获得执行机会后，才在 `15:30:48` 跑起来
- 这次补跑完成后，调度重新对齐到正常节奏，所以后面恢复成 `15:40:00`、`16:00:00`

后续如果再看到这种模式，优先按下面顺序判断：

1. 先看最近一次正常触发后，程序内部下一轮 `60s` 轮询理论上应在什么时候发生
2. 再对照 `pmset -g log` 看机器是在哪个时点真正 `Entering Sleep`，又在哪个时点真正 `Wake from Normal Sleep`
3. 如果第一次补跑发生在唤醒后几十秒内，而之后又恢复到规则 cron 边界，那么应归类为“短 timer 唤醒抖动”，不是“调度逻辑再次长周期漂移”
